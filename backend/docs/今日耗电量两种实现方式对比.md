# 今日耗电量两种实现方式对比

## 一、实现方式概览

| 维度 | 原实现（直接 SUM） | 现实现（LAG 增量） |
|------|-------------------|-------------------|
| **数据来源** | `tunnel_carbon_day_push.power_value` | 同上 |
| **计算方式** | `SUM(power_value) WHERE DATE(upload_time) = 今日` | `今日累计 - 昨日累计`（LAG 窗口函数） |
| **SQL 复杂度** | 单表简单聚合 | CTE + LAG 窗口函数 |
| **查询次数** | 1 次 | 每隧道 1 次（或批量 1 次） |
| **使用场景** | 新接口初版 | `/api/analyze/statistics`（运行 2 年+，甲方认可） |

---

## 二、`power_value` 数据含义

`tunnel_carbon_day_push` 表中 `power_value` 注释为「电表读数」，通常有两种理解：

### 2.1 若为**累计值**（电表总读数，类似 kWh 表盘）

- **原实现（直接 SUM）**：错误  
  - `SUM(power_value)` = 今日所有电表读数之和，表示的是「今日结束时的总读数」，不是耗电量  
  - 耗电量 = 今日读数 - 昨日读数  

- **现实现（LAG）**：正确  
  - `cumulativePowerConsumption = SUM(power_value)` 按日汇总  
  - `dailyPowerConsumption = 今日累计 - LAG(昨日累计)` = 当日耗电量  

### 2.2 若为**日增量**（当日该电表新增耗电）

- **原实现（直接 SUM）**：正确  
  - `SUM(power_value)` = 当日所有电表耗电之和  

- **现实现（LAG）**：也可正确  
  - 若每日一条记录，LAG 差值 = 当日增量，与直接 SUM 等价  

---

## 三、statistics 接口的假设

`/api/analyze/statistics` 使用 LAG 计算 `actualPowerConsumption`，说明其假设为：

- `power_value` 为**累计电表读数**
- 当日耗电量 = 当日累计 - 前日累计  

该逻辑已运行 2 年多且获甲方认可，因此应沿用该假设。

---

## 四、结论与建议

| 结论 | 说明 |
|------|------|
| **数据模型** | 按 statistics 接口，`power_value` 视为累计读数 |
| **计算方式** | 应采用 LAG 增量方式，与 statistics 一致 |
| **原实现** | 在累计模型下会得到错误结果（把总读数当耗电量） |
| **现实现** | 与 statistics 一致，逻辑正确 |

**建议**：今日数据汇总接口继续使用 LAG 增量实现，与 statistics 保持一致，并通过批量查询优化性能。

---

## 五、批量查询实现说明

为避免 N+1（每隧道一次 `getEnergyCarbonVo`），新增 `selectTodayPowerConsumptionBatch`：

- **一次 SQL** 完成所有用户隧道的今日耗电汇总
- 使用 `(devicelist_id, addr) IN (valid_pairs)` 子查询，仅统计用户隧道下的电表
- LAG 逻辑与 `getEnergyCarbonVo` 一致
- **方向过滤**：批量版包含 `tunnel_power_edge_computing` 中所有电表；per-tunnel 版按 `direction`（左线/右线）过滤。若隧道配置了双方向，批量版可能略多，通常更接近真实总耗电
