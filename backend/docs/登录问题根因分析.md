# 非 admin 用户登录失败根因分析

## 现象

- 修改 `userPowerOverview` 接口后，除 admin 外，其他账号（如 test_dev）无法登录
- 报错：`Cannot invoke "com.scsdky.common.core.domain.entity.SysUser.getUserId()" because "user" is null`
- 数据库确认：test_dev 在 `sys_user` 中存在，`sys_user_role` 也有对应记录

## 根因结论

**问题不在 userPowerOverview 本身，而在 `TokenService.getLoginUser()` 的 JSON 二次反序列化逻辑。**

`userPowerOverview` 与登录是不同请求，其修改并未直接改动登录逻辑。真正导致问题的是：**每次带 token 的请求**（包括登录后的 getInfo、dashboard 等）都会经过 `JwtAuthenticationTokenFilter`，从 Redis 取出 LoginUser 时，`TokenService.getLoginUser()` 会做一次多余的 JSON 序列化/反序列化，导致嵌套的 `SysUser`（`LoginUser.user`）在某些用户上反序列化为 null。

## 技术细节

### 1. 问题代码位置

`TokenService.getLoginUser()`（约 57-81 行）：

```java
Object object = redisCache.getCacheObject(userKey);  // Redis 已正确反序列化
String str = JSONObject.toJSONString(object);         // 转 JSON 字符串
JSONObject jsonObject = JSONObject.parseObject(str);  // 解析
jsonObject.remove("@type");                           // 移除根级 @type
LoginUser user = JSONObject.toJavaObject(jsonObject, LoginUser.class);  // 二次反序列化
return user;
```

### 2. 为何会得到 user=null

- Redis 使用 `FastJson2JsonRedisSerializer` 存储 LoginUser，序列化时带有 `@type`，反序列化是正常的。
- `TokenService` 不直接使用 Redis 返回的对象，而是再做一次：`toJSONString` → `parseObject` → `remove("@type")` → `toJavaObject`。
- FastJSON 对**嵌套对象**的反序列化存在已知问题：复杂嵌套（如 SysUser 的 roles、dept）在二次反序列化时可能失败，将对应字段设为 null。
- admin（userId=1）的 SysUser 结构相对简单（如 dept、roles 较少），反序列化成功；test_dev 等用户结构更复杂，嵌套反序列化失败，导致 `user` 为 null。

### 3. 为何会“污染” Redis

`JwtAuthenticationTokenFilter` 中：

```java
tokenService.verifyToken(loginUser);  // 若 token 快过期，会调用 refreshToken
```

`verifyToken` 会调用 `refreshToken(loginUser)`，把当前 `loginUser` 写回 Redis。若此时 `loginUser.user` 已为 null（来自有问题的二次反序列化），就会把 `user=null` 的 LoginUser 覆盖写入 Redis，造成后续请求持续拿到错误的 LoginUser。

### 4. 为何表现为“修改 userPowerOverview 之后才出现”

- 修改前：可能较少触发 dashboard / userPowerOverview 等接口，或请求顺序不同，问题暴露较少。
- 修改后：新前端或新流程在登录后更早、更频繁地请求 dashboard，从而更早触发 `getLoginUser` 的二次反序列化和 `verifyToken`，问题被放大。

本质是：**问题一直存在于 TokenService 的二次反序列化逻辑，只是在新流程下更容易被触发。**

## 修复方案

**直接使用 Redis 返回的 LoginUser，去掉多余的 JSON 二次反序列化。**

Redis 的 `FastJson2JsonRedisSerializer` 已经正确完成反序列化，无需再做一次 `toJSONString` + `toJavaObject`。去掉这段逻辑即可避免嵌套 SysUser 被错误置为 null。
